Peter Harding’s Summary

PerformIQ has been implementing stress and volume tests on large commercial systems since 1990. Most recently, these projects have involved testing of new and upgraded applications prior to production roll-out.

PerformIQ has been involved in a number of large stress testing projects using commercial GUI tools (eg Mercury Interactive's LoadRunner and Rational's Performance Studio/Robot). More often than not been involved in projects where commercially available tools could not handle the underlying protocols and we found various ways around these limitations using our own and OpenSource tools.

The approach we normally recommend for system testing of distributed architectures is to split the client into a GUI layer and a network layer with an appropriately defined API between these (passing data structures with the various application data required). If this is done, the front-end GUI can be peeled off and robots implemented in C, C++, Java or a scripting language such as Python or PERL, which use the API and the bottom end of the client to construct a robot which can be run from the test-harness. Typically, this may only be possible without some degree of re-engineering where the client architecture has been developed with this in mind. Generally, this is an approach we recommend when we are involved early enough in a development project, as an approach like this can produce significant savings in time and effort required to implement a stress testing environment.

PerformIQ has built up expertise in the areas of stress testing, load balancing and capacity planning over many years. This expertise includes development of a suite of software testing tools that can be used to add value to the particular assignment for which we are engaged. PerformIQ can be used to constitute an infrastructure testing team for large software developments, for validation of hardware purchases or simply to audit and field test hardware and network requirements.
