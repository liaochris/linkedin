Huw Nichols' Projects

  • Price comparison engine and site for corporate stationary purchases

    • December 2011 to Present
    Team Members: Huw Nichols

    Scope: The project was to build a price comparison site, including backend and data import, aimed at the UK schools and business stationary markets. Such is the nature of ordering stationary for a business, the order is usually for many different items at the same time. The system had to allow for this and perform the actual comparisons not on individual products but rather on the users complete order.

    Skills: nodejs, express, nowjs, websockets, data analysis, systems design, server provisioning, training, graphics creation, javascript, HTML5, CSS3, cross browser, adaptive design, web scraping, dustjs, AMD, site optimisation.

    The project took approx. 6 weeks from December 2011..

    Data design:
    Supplier data feeds often have exactly the same product but with differing part numbers and textual descriptions. Solution turned out to be both easy and hard at the same time — use the international bar code for the product. Snag, not all data feeds provide this information or it is inaccurate.

    The data import and merging routines cope with this complexity as well as the usual addition and removal of product records from the feeds as suppliers update stock holdings.

    The comparison engine is a nodejs service with mongodb as the backing store.

    Front end:
    My client was breaking into an existing market and really wanted to impress the suppliers and grab their attention. I proposed a single page web app with real time data via websockets (nowjs providing the glue).

    Results were to be displayed ranked by suppliers who could provide the largest portion of the entire order and suggestions as to where to get the rest at best price and minimising suppliers.

    The finished system displayed comparison results in real time as each new item was added to the cart. Total number of items in catalog ~ 350K.

    Sadly the project failed soon after my client launched it due to their running out of money to fund further development.

    Whilst specific urls are not available due to NDAs etc. some of these interfaces can still be seen in use today.

  • Real Paper

    • December 2011 to Present
    Team Members: Huw Nichols

    Scope: Imagine, for a moment, the feel and experience of a traditional quality paper publication. Now imagine that same publication on the web — not even close is it? Busy sites, adverts, clumsy navigation, slow page loading and that's not even thinking of the backend problems for the publisher and content providers.

    Now go back to the quality print publication, imagine having that same feeling of quality, the same familiar layout and ease of reading, the ability to flick through the pages rather than a clumsy menu system, searchable, indexable, zoomable, usable. Imagine all that plus seamless display on multiple devices, almost any modern browser in fact, from a single data and presentation set. Standards compliant HTML5, CSS3, Javascript. Advanced backend for ease of publication, write once display anywhere, but display it so it retains that certain something. The quality feel.

    This is a relatively long term project is to prototype such a system and is currently in it's early alpha stages. It is a proof of concept in early stages.

    Skills: nodejs, coffeescript, javascript, mongodb, noSQL, CSS3, HTML3, CSS frameworks (boostrap, less 4), mobile app, design, data structures, GUI design, user experience flow, creative writing, project management, funding, redis, cache systems, website seo and optimization

    The test site (when available) is at http://strutitmagazine.com

    This is the start of a fashion magazine using the alpha as a delivery platform. We chose a fashion magazine as the first test because the print versions, think Cosmopolitan, Vogue etc., have many of the quality characteristics common to any target publication. They are also periodicals and thus provide an opportunity to work on archiving, cross issue navigation and so on.

Prospero3

2005 – August 2009 (4 years) Remote

Develop real time browser based interfaces for the Sports and Spread betting industries.

The owner of Ariel Communications sold the company and started Prospero3 in the same market place thus my work here was essentially a continuation of the job I held at Ariel.

This time round we moved over to using Flex as the main development tool for the client side work. However, the MXML approach to working with Flex did not meet our needs, neither did the built in components provided with the framework. The reasons being:

• Our systems had to be fast and have a small download footprint at the client side
• The interfaces had to match as closely as possible (to the pixel sometimes) the designs produced by the clients graphic designers — often times this would simply have been impossible to match using standard components of any framework then or now existing.

I therefore perfected a workflow allowing direct hand coding of all interface elements (avoiding MXML) and at the same time created our own event driven framework and custom highly visually configurable components. It is this framework that we used and enhanced over the time I worked with Prospero3.

The framework was designed from the outset to be reasonably future proof and in some cases client expandable so it is pleasing to see the durability of the solutions I created allow many of our clients to continue to use them to this day. Sure, some will have had features added but on checking I find a number are clearly still using the main base I created 2-4 years ago.
