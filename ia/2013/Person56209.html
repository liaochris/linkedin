Senior Software Engineer

ScaleDB

November 2008 – November 2009 (1 year 1 month)

One member of a small team of software engineers working at a startup developing a Clustered Mysql storage engine. The storage engine handles the database indexing, buffer manager management, locking, logging, and file IO and replaces the default MySQL storage engine. Key features of the ScaleDB storage engine include shared-disk clustering which allows MySQL servers to be running on many different machines and all share the same data without partitioning. Nodes can be brought up and down dynamically allowing for High-Availability, and Scalability.

Coding is multi-threaded C and C++ for a Linux target platform. My largest project was the development of the distributed locking manager server infrastructure. The distributed lock manager is a standalone daemon that all the MySQL servers communicate with using tcpip/ip sockets with a custom message protocol.

Implemented abstractions and did performance testing on low level system functions including threading, condition variables, sockets, mutexes, read-write locks, spin locks, and disk IO.

Did performance testing to identify bottlenecks in the code and resolved the bottlenecks found. Bottlenecks found included redundant disk IO which caused order of magnitude performance penalties and in-appropriate usage of mutexes which are expensive to lock and unlock. Testing and retesting of performance was a multi-week process where bottlenecks are found and fixed. Tools used for performance testing include gprof, iostat, vmstat, and custom code instrumentation.

Troubleshooting, debugging, and fixing of over 200 bugs that were filed by the QA team and all parts of the system including the index, log manager, buffer manager, and the interface between the storage engine and MySQL.

Software Engineer

Yahoo

Public Company; 10,001+ employees; yhoo; Internet industry

February 2007 – November 2008 (1 year 10 months)

Work on Yahoo’s custom ETL software written in C, C++ and perl that is used to collect and parse all of the logs from Yahoo’s web servers and other applications such as Yahoo Messenger and Yahoo Toolbar. The software runs on over 400 BSD Unix and Linux compute nodes and is continually processing incoming log files. Log files are parsed and event information is recorded into a multi-terabyte data warehouse.

Significant accomplishments include reduction of completion time for end of day processing from 8AM to 2AM while also accommodating for increased data volume of over 100% during the same period.

Numerous bottlenecks were found and resolved including excessive disk IO resolved by reducing the number of IO transactions done, to prefer IO transactions to locations with faster performance, increasing the size of the average IO operation, and tuning compression levels to increase CPU usage and decrease IO. Memory footprint of the processes was reduced in order to run more parallel processes on 8 core boxes. CPU was reduced by modifying the algorithm used for merging from an n-squared to log-n algorithm. Network topology was modified to provide increased throughput to choke points when found. Hardware upgrades were made to accommodate additional capacity across processing boxes and storage.

Additionally the failure rate of the daily processes was reduced by fixing issues that caused core dumps such as buffer overflows or exhausting the memory space of the process.

Was responsible for porting the applications and core libraries from BSD to linux and for incorporating Yahoo standard build infrastructure into our projects to ensure that builds for our applications were done consistently by all developers.
(note: converted from contractor to fulltime in July, 2007)

Senior Software Engineer

Etrade Financial

Public Company; 1001-5000 employees; ETFC; Financial Services industry

July 2006 – February 2007 (8 months)

Maintained and enhanced the Smart Alerts© and Advanced Orders© web services in C running on Linux (Red Hat 7.2, Red Hat AS 3).

Memory usage reduction for the smart alerts services. When I took over this code base, three boxes were used to handle the load, and the 4GB on each box was utilized plus swap. After profiling and refactoring, 2GB were used on each box with 2GB of spare memory free.

Rewrite of the bulk alert sender applications to allow support staff to send millions of alerts to customers with less time delay (6 x speed improvement) and with more diagnostic information to allow support staff to identify which alerts failed and for what reason.

Creation of utility classes that were contributed to the companies shared code base including 1) a smart pointer template, similar to Boost’s scoped pointer (non-ownership transferring), that used free to deallocate memory instead of delete. This was useful with the vast internal API’s that allocated memory using malloc instead of new. 2) A template class used to wrap an aggregation web service, which allowed multiple web services to be called with one service call.

Running of Valgrind on the smart alerts code to eliminate memory errors such as free calls on memory allocated with new and other memory corruptions.

Making the smart alerts codebase compile cleanly without warnings.

Software Engineer

Mako Global Derivatives

Privately Held; 51-200 employees; Financial Services industry

February 2002 – June 2006 (4 years 5 months)

Coding and support of a real-time trading system that consists of 1 million lines of multi-threaded C++ code and runs as a distributed client server system across a network of over 200 sun servers.

Worked on the automated trading applications that subscribe to multiple servers in order to obtain current prices and theoretical valuation for each product and send inserts and updates of orders every time the valuation changes to our internal order book server. From our order book server, orders are routed to an application that converts orders from Mako's order format to the exchanges' format and then sends the orders to the exchange using their APIs. Millisecond performance and deterministic behavior is critical to avoid trading at bad prices.

Ported the codebase from the Sun compiler running on Sparc to GCC compiler running on AMD. We used gcc 3.3.6 and had to fix both compiler errors found by gcc and run-time logic errors in the system that caused erratic behavior on the new platform. For example we had many cases were pointers were casted to integers and used in appropriate ways.

Numerous (over 100) bug fixes and performance enhancements released to production. Fixes included incorrect calculations, race conditions, locking issues, memory corruption. One performance improvement was eliminating regex comparisons that were being done upon the receipt of every heartbeat message by caching the results of the first heartbeat from each host and pid combination. This regex issue was identified using gprof and Quantify. Inappropriate use of strings, also frequently caused performance issues. Numerous locking issues were caused by applications grabbing two locks in different orders on different threads causing A-B B-A deadlocks. Memory corruption issues were caught with Valgrind and Purify, an example of which was char buffers that were being overrun in fraction to integer conversions for display.
